## Executive Summary

Verocious Motorsports is **failing Core Web Vitals** on both mobile and desktop, with 456 poor mobile URLs and 448 poor desktop URLs in Google Search Console — zero URLs currently rated "Good." The two failing metrics are **LCP** (3.3s mobile / 3.0s desktop) and **CLS** (0.75 mobile / 0.25 desktop). INP is the only passing metric (133ms / 110ms).

The root causes fall into three categories: (1) the LCP element on mobile is a CSS `background-image` that the browser cannot preload or prioritize, combined with 9 render-blocking scripts totaling 1,271 KB that delay paint by 2+ seconds; (2) the cookie consent banner is statically positioned and pushes content down, causing 0.2+ CLS on every page load; and (3) massive third-party script redundancy — both client-side and server-side GTM are running simultaneously, with the old GA4 direct integration and client-side Facebook Pixel still loading despite server-side handling all tracking.

This report contains **18 prioritized fixes** across 3 phases. Phase 1 (quick wins) targets the highest-impact items that can be deployed in 1-2 days. Full implementation should reduce LCP below 2.5s and CLS below 0.1 on both mobile and desktop.

#### Fix #1 — Fix non-www to www redirect (780ms mobile penalty)

**Fix ID:** fix-001
**Metric:** LCP | **Impact:** 9.5 | **Effort:** 2 | **Pages:** All

**Problem:**
`verociousmotorsports.com` redirects to `www.verociousmotorsports.com`, adding 780ms on mobile and 230ms on desktop to every page load. PSI flags this under "Avoid multiple page redirects" with 584ms redirect duration measured in lab.

**Evidence from PSI scan:**
- Mobile: redirects audit shows 780ms savings
- Desktop: redirects audit shows 230ms savings
- Document latency: 584ms redirect duration

**Solution — Option A (DNS-level, preferred):**

If using Cloudflare or another CDN with edge rules:

```
# Cloudflare Page Rule
Match: verociousmotorsports.com/*
Setting: Forwarding URL (301)
Destination: https://www.verociousmotorsports.com/$1
```

This moves the redirect to the CDN edge, reducing it from 780ms to ~10-50ms.

**Solution — Option B (DNS CNAME):**

Point the bare domain `verociousmotorsports.com` directly to the same server as `www.verociousmotorsports.com` via an `A` record or CNAME flattening, so both resolve to the same origin without a redirect.

```
# DNS Records
verociousmotorsports.com    A       <same IP as www>
www.verociousmotorsports.com CNAME  <hosting provider>
```

**Solution — Option C (If NetSuite controls DNS):**

Contact NetSuite support to configure the bare domain to serve content directly without redirecting, or to add the redirect at the CDN/load balancer level rather than at the application level.

**Expected improvement:** 580-780ms LCP reduction on mobile, 200-230ms on desktop.

**Verification:** Run PSI scan and confirm "Avoid multiple page redirects" audit no longer appears, and document latency shows 0 redirect duration.

#### Fix #2 — Convert LCP element from CSS background-image to `<img>` tag

**Fix ID:** fix-002
**Metric:** LCP | **Impact:** 9.0 | **Effort:** 3 | **Pages:** Homepage

**Problem:**
The mobile LCP element is `div.home-infoblock.home-infoblock0` with a CSS `background-image`. This pattern is invisible to the browser's preload scanner, meaning the browser cannot start fetching the image until CSS is fully parsed and CSSOM is built. PSI shows 2,014ms element render delay on mobile.

**Evidence from PSI scan:**
- LCP element: `<div class="home-infoblock home-infoblock0" style="background-image: url('...vms_homethumb_tubing_v3.jpg')">`
- Element render delay: 2,014ms (mobile)
- PSI explicitly flags: "fetchpriority=high should be applied"
- LCP request not discoverable by preload scanner because it's a CSS background

**Current code (SCA template):**
```html
<div class="home-infoblock home-infoblock0"
     style="background-image: url('/site/img/Backgrounds/vms_homethumb_tubing_v3.jpg')">
  <span class="home-infoblock-text">SHOP TUBING</span>
</div>
```

**New code:**
```html
<div class="home-infoblock home-infoblock0">
  <img src="/site/img/Backgrounds/vms_homethumb_tubing_v3.jpg"
       alt="Shop Tubing"
       width="382"
       height="160"
       fetchpriority="high"
       decoding="async"
       class="home-infoblock-image">
  <span class="home-infoblock-text">SHOP TUBING</span>
</div>
```

**CSS changes required:**
```css
/* Before */
.home-infoblock {
  background-size: cover;
  background-position: center;
  /* ... other styles */
}

/* After */
.home-infoblock {
  position: relative;
  overflow: hidden;
  /* Remove background-image, background-size, background-position */
}

.home-infoblock-image {
  position: absolute;
  top: 0;
  left: 0;
  width: 100%;
  height: 100%;
  object-fit: cover;
  object-position: center;
}

.home-infoblock-text {
  position: relative; /* Ensure text renders above image */
  z-index: 1;
}
```

**Where to find this in the codebase:**
- SCA template file: Look in the theme's `Modules/` directory for the homepage module (likely `Home@X.X.X` or `HomePage@X.X.X`)
- The template will be a `.tpl` (Handlebars) file containing the `home-infoblock` markup
- CSS: Look in the theme's SCSS files under `Modules/Home*/` or `Workspace/Extras/`

**Expected improvement:** 1,500-2,000ms LCP reduction on mobile by making the image discoverable to the preload scanner and allowing `fetchpriority="high"` to work.

**Verification:** Run PSI mobile scan. LCP element should now show as an `<img>` tag. Element render delay should drop significantly. "fetchpriority=high" audit should pass.

#### Fix #3 — Fix cookie consent banner CLS (position: fixed)

**Fix ID:** fix-009
**Metric:** CLS | **Impact:** 9.5 | **Effort:** 1 | **Pages:** All
**Status:** Client reports this is in progress — verify current implementation.

**Problem:**
The cookie consent banner is statically positioned and pushes page content down when it appears, causing ~0.2+ CLS. This is the single largest CLS contributor, explaining why field CLS is 0.75 on mobile while lab CLS is 0 (lab doesn't capture this late-appearing banner).

**Evidence:**
- Field CLS: 0.75 mobile (POOR), 0.25 desktop (NEEDS IMPROVEMENT)
- Lab CLS: 0 mobile, 0.02 desktop (GOOD)
- GSC: 456 poor mobile URLs
- The discrepancy between lab and field proves the CLS comes from post-load dynamic content

**Fix — CSS override:**
```css
/* Cookie consent banner — fix CLS by overlaying instead of pushing content */
.cookie-consent-banner,
.cc-banner,
.cc-window,
[class*="cookie-consent"],
[class*="cookie-banner"] {
  position: fixed !important;
  bottom: 0 !important;
  left: 0 !important;
  right: 0 !important;
  z-index: 9999 !important;
  /* Remove any top/margin properties that would push content */
  top: auto !important;
  margin: 0 !important;
}
```

**Where to apply:** Add to the theme's main SCSS file or as a standalone CSS override in the NetSuite backend. Identify the exact cookie banner class by inspecting the live site, then narrow the selector to the actual class used.

**Expected improvement:** CLS reduction of ~0.2+ on all pages. This single fix should bring desktop CLS below 0.1 (passing) and significantly improve mobile CLS.

**Verification:** After deploying, wait for field data to update in CrUX (28 days). In the meantime, use Chrome DevTools Performance tab to verify no layout shifts from the banner. Run `new PerformanceObserver(l => l.getEntries().forEach(e => console.log('CLS:', e.value, e.sources))).observe({type: 'layout-shift', buffered: true})` in the console.

#### Fix #4 — Add `<link rel="preload">` for LCP image

**Fix ID:** fix-003
**Metric:** LCP | **Impact:** 8.5 | **Effort:** 1 | **Pages:** Homepage

**Problem:**
The LCP image has no preload hint. Even before converting to `<img>` (Fix #2), adding a preload tells the browser to start fetching the image immediately during HTML parsing rather than waiting for CSS/JS to discover it.

**Fix — Add to `shopping.ssp` `<head>`:**
```html
<!-- Preload LCP image for homepage -->
<link rel="preload"
      as="image"
      href="/site/img/Backgrounds/vms_homethumb_tubing_v3.jpg"
      fetchpriority="high">
```

**Where to apply:** In the SCA theme, find `shopping.ssp` (the main SSP file for the shopping touchpoint). Add this `<link>` tag inside the `<head>` section, ideally as early as possible (right after `<meta charset>`).

**Note:** If implementing Fix #2 first (converting to `<img>` with `fetchpriority="high"`), this preload becomes less critical but still beneficial as the preload starts the fetch before the HTML parser reaches the `<img>` tag.

**Expected improvement:** 200ms+ resource load delay reduction.

#### Fix #8 — Add preconnect hints

**Fix ID:** fix-010
**Metric:** LCP | **Impact:** 6.0 | **Effort:** 1 | **Pages:** All

**Problem:**
Zero preconnect hints on the page. The browser must discover third-party origins through resource loading, then perform DNS lookup + TCP connect + TLS handshake before fetching. Preconnect eliminates this cold-start penalty.

**Fix — Add to `shopping.ssp` and `checkout.ssp` `<head>` (as early as possible):**
```html
<!-- Preconnect to critical third-party origins -->
<link rel="preconnect" href="https://fonts.googleapis.com">
<link rel="preconnect" href="https://fonts.gstatic.com" crossorigin>
<link rel="preconnect" href="https://www.googletagmanager.com">
<link rel="preconnect" href="https://sst.verociousmotorsports.com">
```

**Note:** Only preconnect to origins you know will be used on every page load. The above four are guaranteed on every page. Do NOT add preconnect for origins that are conditionally loaded (e.g., Facebook if you're removing the pixel).

**Expected improvement:** 100-300ms saved per origin on first page load (DNS + TCP + TLS).

#### Fix #9 — Add font-display: swap to @font-face rules

**Fix ID:** fix-011
**Metric:** LCP | **Impact:** 5.5 | **Effort:** 1 | **Pages:** All

**Problem:**
16 of 24 `@font-face` rules are missing the `font-display` property. This causes Flash of Invisible Text (FOIT) — the browser hides text until the font downloads, delaying text paint and potentially impacting LCP if text is the LCP element.

**Fix — Add `font-display: swap` to all @font-face rules in theme SCSS:**
```scss
// Find all @font-face declarations in the theme SCSS and add font-display: swap
// Common locations: _fonts.scss, _variables.scss, or main stylesheet

@font-face {
  font-family: 'DroidSans';
  src: url('/site/fonts/DroidSans.woff2') format('woff2');
  font-weight: normal;
  font-style: normal;
  font-display: swap; /* ADD THIS LINE */
}

@font-face {
  font-family: 'DroidSans';
  src: url('/site/fonts/DroidSans-Bold.woff2') format('woff2');
  font-weight: bold;
  font-style: normal;
  font-display: swap; /* ADD THIS LINE */
}

@font-face {
  font-family: 'OpenSans';
  src: url('/site/fonts/OpenSans-SemiBold.woff2') format('woff2');
  font-weight: 600;
  font-style: normal;
  font-display: swap; /* ADD THIS LINE */
}

// Repeat for ALL @font-face declarations
// Also add to any Google Fonts loaded via @import or <link>
```

**For Google Fonts loaded via `<link>`:**
```html
<!-- Add &display=swap to Google Fonts URLs -->
<link href="https://fonts.googleapis.com/css2?family=Exo:wght@400;700&family=Nunito+Sans&family=Oswald&family=Poppins&family=Nunito&display=swap" rel="stylesheet">
```

**Where to find @font-face rules in SCA:**
- Theme SCSS files in `Workspace/` directory
- `Modules/` — check for font-related modules
- `ns_packages/` — some fonts may be in extension packages
- Run `grep -r "@font-face" --include="*.scss" --include="*.css"` in the theme directory

**Expected improvement:** Eliminates FOIT (text becomes visible immediately with fallback font, then swaps when custom font loads).

#### Fix #10 — [CHECKOUT] Fix reCAPTCHA loading 4 times

**Fix ID:** fix-016
**Metric:** LCP | **Impact:** 9.0 | **Effort:** 2 | **Pages:** Checkout

**Problem:**
reCAPTCHA is loaded **4 separate times** on the checkout page:
- 2x blocking from `google.com/recaptcha/api.js` (render-blocking, in `<head>`)
- 2x async from `gstatic.com/recaptcha/releases/` (async, but still redundant)

This is a bug in the **BluePoint.ReCaptcha** extension.

**Fix:**
1. Open the BluePoint.ReCaptcha extension source code in the SCA theme
2. Find where `recaptcha/api.js` is loaded — there will be duplicate script injections
3. Deduplicate to a single async load:

```html
<!-- Should load ONCE, and ASYNC — not blocking -->
<script async src="https://www.google.com/recaptcha/api.js?render=<SITE_KEY>"></script>
```

**Steps to fix in BluePoint.ReCaptcha:**
1. In the extension's entry point module, find the script loader
2. Add a guard to prevent double-loading:
```javascript
// Guard against duplicate reCAPTCHA loads
if (!window._recaptchaLoaded) {
  window._recaptchaLoaded = true;
  var script = document.createElement('script');
  script.src = 'https://www.google.com/recaptcha/api.js?render=' + siteKey;
  script.async = true;
  // script.defer = true;  // optional, if not needed immediately
  document.head.appendChild(script);
}
```

3. Remove any `<script>` tags for reCAPTCHA from the `checkout.ssp` template `<head>` — it should only load via the extension's JS

**Expected improvement:** ~790 KiB wasted JavaScript eliminated from checkout.

#### Fix #12 — Defer Klaviyo SDK loading

**Fix ID:** fix-007
**Metric:** LCP/TBT | **Impact:** 6.5 | **Effort:** 2 | **Pages:** All

**Problem:**
7 Klaviyo Custom HTML tags in client GTM. The SDK Loader fires on all pages on `All Pages` trigger, loading ~160 KiB of JavaScript synchronously during page load. Klaviyo consumes 88.8ms main thread time on mobile.

**Fix — In GTM (GTM-57T4T5BW):**

   - **Trigger Type:** Custom Event
   - **Event Name:** `klaviyo_load`
   - Fire on: All Custom Events matching `klaviyo_load`

3. Create a new tag to fire the custom event after delay:
   - **Tag Name:** "Klaviyo Deferred Loader"
   - **Tag Type:** Custom HTML
   - **HTML:**

```html
<script>
```

   - **Trigger:** All Pages (fires early, but Klaviyo SDK only loads 3s after page load)

4. The other 6 Klaviyo tags (Add To Cart, Newsletter Signup, etc.) depend on the SDK being loaded, so they'll naturally defer too — no changes needed to them.

**Expected improvement:** ~160 KiB deferred from initial load, reduced TBT.

#### Fix #15 — Preload critical fonts

**Fix ID:** fix-015
**Metric:** LCP | **Impact:** 5.0 | **Effort:** 1 | **Pages:** All

**Problem:**
Three heavily-used self-hosted fonts load with no preload hint and only 2-hour cache TTL:
- DroidSans.woff2 (58 KiB)
- DroidSans-Bold.woff2 (61 KiB)
- OpenSans-SemiBold.woff2 (43 KiB)

**Fix — Add to `shopping.ssp` and `checkout.ssp` `<head>`:**
```html
<!-- Preload critical fonts -->
<link rel="preload" as="font" href="/site/fonts/DroidSans.woff2" type="font/woff2" crossorigin>
```

#### Fix #17 — Increase cache TTL from 2hrs to 1yr for versioned assets

**Fix ID:** fix-013
**Metric:** LCP | **Impact:** 6.0 | **Effort:** 3 | **Pages:** All

**Problem:**
All first-party JS, CSS, and fonts are cached for only **2 hours** (`max-age=7200`). These files already have cache-busting timestamps (`?t=1770834117652`), so they can safely be cached for much longer.

**Evidence from PSI scan:**
- 1,640 KiB wasted on repeat visits due to short cache TTL
- Top offenders: shopping.js (366 KB), shopping_5.js (336 KB), shopping-templates_5.js (250 KB), shopping_5.css (204 KB), FontAwesome (197 KB), fonts (218 KB combined)

**Fix — Set cache headers:**
```
Cache-Control: public, max-age=31536000, immutable
```

**How to implement (depends on hosting setup):**

**Option A — If using Cloudflare or CDN:**
Create a page rule or cache rule:
```
# Match versioned static assets
URL pattern: *verociousmotorsports.com/*.js?t=*
URL pattern: *verociousmotorsports.com/*.css?t=*
URL pattern: *verociousmotorsports.com/site/fonts/*
Action: Cache-Control → public, max-age=31536000, immutable
```

**Option B — If NetSuite controls headers:**
This may require a NetSuite support ticket to change the default `Cache-Control` header for static assets from `max-age=7200` to `max-age=31536000`. Reference that the assets already have cache-busting query strings.

**Option C — If using a web server (Apache/Nginx) in front:**
```apache
# Apache .htaccess
<IfModule mod_expires.c>
  ExpiresActive On
  ExpiresByType application/javascript "access plus 1 year"
  ExpiresByType text/css "access plus 1 year"
  ExpiresByType font/woff2 "access plus 1 year"
</IfModule>
```

**Expected improvement:** 1,640 KiB saved on repeat visits. Returning visitors will load the site significantly faster.
