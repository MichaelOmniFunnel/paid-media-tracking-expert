# WooCommerce Conversion Tracking Implementation Guide

This guide covers end-to-end conversion tracking for WooCommerce stores, from plugin-based GTM integration through full server-side measurement via Stape.io.

---

## 1. GTM on WooCommerce

### Plugin Selection

GTM4WP (free, by Thomas Geiger) is the standard for most WooCommerce setups. It auto-injects the GTM snippet and generates GA4 ecommerce dataLayer events out of the box.

For detailed plugin configurations and manual installation code, read references/plugin-configs.md

### Key Configuration Decisions

- **Container code placement:** Use codeless injection via wp_head/wp_body_open hooks
- **Product ID format:** Must match your product feed (SKU vs WordPress post ID)
- **WooCommerce integration:** Enable all event tracking options
- **Customer data on order received:** Yes, for enhanced conversions

## 2. DataLayer for WooCommerce

### Events Generated by GTM4WP

GTM4WP generates standard GA4 ecommerce dataLayer events: `view_item`, `view_item_list`, `add_to_cart` (both standard and AJAX), `begin_checkout`, `add_shipping_info`, `add_payment_info`, and `purchase`.

### Variable Product Handling

Variable products present a common tracking challenge. The parent product has one ID while each variation has its own ID and potentially its own SKU. Ensure the Product ID format in GTM4WP matches your feed.

GTM4WP updates the dataLayer when a customer selects a variation. Verify with the dataLayer debugger that item_id and item_variant change on selection.

For complete dataLayer payloads, user data layer code, and variation listener, read references/datalayer-examples.md

## 3. Server-Side via Stape

### Architecture

```
Browser (WooCommerce site)
  |
  v
Client-side GTM container (web)
  |  sends events via GA4 transport to first-party subdomain
  v
Stape.io server container (sGTM)
  |
  +---> Meta Conversions API
  +---> TikTok Events API
  +---> Google Ads Enhanced Conversions
  +---> GA4 (server-side)
```

### Setup Steps

1. Create a Stape server container at stape.io
2. Set up a first-party subdomain (CNAME record, e.g., sst.clientdomain.com)
3. Update client-side GTM GA4 tag with server_container_url
4. Configure server container tags (Meta CAPI, TikTok Events API, Google Ads, GA4)

### WooCommerce Webhook Approach

WooCommerce supports webhooks natively (WooCommerce > Settings > Advanced > Webhooks). Create a webhook with topic "Order created" pointing to a Stape endpoint. The payload includes full order details, customer data, and line items.

For server-side plugin options, read references/plugin-configs.md

## 4. Meta CAPI on WooCommerce

### Architecture Options

| Approach | Best For | Complexity |
|---|---|---|
| Facebook for WooCommerce plugin | Single pixel, minimal customization | Low |
| PixelYourSite Pro | Plugin-based CAPI without GTM | Medium |
| Stape server-side (recommended) | Multi-platform, full control | Medium-High |
| Manual PHP implementation | Maximum control, custom events | High |

### Stape Server-Side Approach (Preferred for OFM)

1. GTM4WP generates the dataLayer on the WooCommerce frontend
2. Client-side GTM fires GA4 events to the Stape server container
3. Meta CAPI tag (Stape template) sends events to Meta
4. Configure with Pixel ID, Access Token, event mapping, user data, custom data

For manual CAPI PHP code, plugin details, and checkout field capture code, read references/code-implementations.md

## 5. Click ID Capture

### Strategy

Capture gclid, fbclid, ttclid, and msclkid from URL parameters, store as first-party cookies (90 day expiry), then pass through to WooCommerce order meta via hidden checkout fields.

This enables offline conversion uploads, CAPI events with fbp/fbc for Meta and gclid for Google, and attribution reports in WooCommerce.

For click ID capture JavaScript, hidden checkout field PHP code, and cookie population scripts, read references/code-implementations.md

## 6. Common WooCommerce Tracking Issues

### Cache Plugins Breaking DataLayer

Page caching serves static HTML, causing the dataLayer to contain stale or incorrect data.

| Cache Plugin | Fix |
|---|---|
| **WP Rocket** | Add /checkout/* and /cart/* to "Never Cache" URLs |
| **W3 Total Cache** | Exclude /checkout/order-received from page cache |
| **LiteSpeed Cache** | Add /checkout and /cart to "Do Not Cache URIs" |
| **Cloudflare** | Page Rule: *domain.com/checkout/* -> Cache Level: Bypass |
| **WP Super Cache** | Enable "Don't cache pages with GET parameters," exclude WooCommerce pages |

**Critical:** The /checkout/order-received/ page must NEVER be cached.

### AJAX Add-to-Cart Events Not Firing

Root causes: theme overrides default AJAX handler, non-standard cart template, custom mini-cart JavaScript. Debug by checking browser console for dataLayer.push with add_to_cart on product page click.

### Order Received Page Caching / Duplicate Purchases

GTM4WP includes a built-in flag (orderIdSent) using sessionStorage to prevent duplicate fires. For additional protection, use a custom JavaScript variable in GTM that checks sessionStorage before allowing conversion tags to fire.

### Variable/Grouped Product ID Mismatches

Check the feed format (parent IDs, variation IDs, or SKUs) and ensure the dataLayer item_id matches. Common mismatch: feed uses a prefix format but dataLayer pushes raw IDs.

### Payment Gateway Redirect Tracking Gaps

When customers pay via PayPal, Klarna, or other redirects, the session may be lost. Solutions: WooCommerce webhook as backup, Stape server-side tracking, or PHP order status hook.

### WordPress Security Plugins Blocking Tracking

Common offenders: Wordfence, Sucuri, iThemes Security, All-in-One Security. Whitelist googletagmanager.com, google-analytics.com, connect.facebook.net, analytics.tiktok.com, and the Stape subdomain.

### WooCommerce Blocks (New Checkout)

WooCommerce is transitioning to Blocks-based checkout (React-based). jQuery hooks do not fire. GTM4WP has added Blocks support but verify events fire on all steps. Custom checkout field injection requires the Checkout Block extensibility API instead of PHP hooks.

**Recommendation:** Test thoroughly after any WooCommerce or GTM4WP update, especially if the client migrates from classic to Blocks checkout.
